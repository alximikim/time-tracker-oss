# TimeTracker

Трекер рабочего времени для сотрудников (в т.ч. удалённых) — без Excel и без
Google Sheets. Вся база данных хранится у вас: на своём компьютере или на
своём сервере. Время засчитывается только пока сотрудник сам держит сессию
"начатой" — включённый компьютер сам по себе ничего не значит. Сессия
закрывается либо кнопкой "Закончить работу", либо автоматически при
выключении/выходе из Windows.

## Зачем этот проект

Написан через [Claude Code](https://claude.com/claude-code) — сначала чисто
для себя, чтобы закрыть один конкретный вопрос: сколько часов сегодня
отработал сотрудник, без того, чтобы каждый день кто-то это спрашивал или
заполнял таблицу руками.

Смотрел в сторону Kimai и похожих систем — не подошло не потому, что они
плохие, а потому, что это уже полноценные тайм-трекеры с проектами,
задачами, тайм-шитами и ролями. Мне же было нужно ровно одно: чтобы
сотрудник видел в трее кнопку, нажимал "Начать работу" — и на этом всё, ни
регистрации, ни настроек, ни обучения. Вся сложность (сервер, конфиг,
токены) — на стороне админа; у сотрудника — одна кнопка и больше ничего.

Отсюда и все решения внутри: без Razor и JS-фреймворков в админке, сырой
ADO.NET вместо ORM, HTML собирается C#-строками. Не потому что так "модно",
а потому что для инструмента на несколько человек это самый короткий путь
от кода до работающей вещи, которую можно целиком удержать в голове и
починить за один вечер, если что-то сломалось.

**Кому подойдёт:**
- Небольшой команде/бизнесу, которому нужен только факт учёта часов — без
  проектов, тайм-шитов и биллинга поверх.
- Тем, кто хочет держать данные у себя (свой ПК или VPS), а не в чужом
  облаке.
- Тем, кому надоели Excel/Google Sheets, которые сотрудники забывают или
  ленятся заполнять.

**Кому не подойдёт:** если нужны проекты/задачи/биллинг/отчётность по
клиентам — это не тот инструмент, для этого и существует Kimai и подобные.

Проект открытый, но поддерживается по мере личной необходимости, а не по
SLA — issues и PR приветствуются, но не ждите мгновенной реакции.

Весь код и этот текст, включая раздел, который вы сейчас читаете, написаны
[Claude Code](https://claude.com/claude-code) — без притворства, что это
не так.

## Архитектура

```
[ Клиент: трей-приложение, Windows ]
        |  локальная очередь (SQLite, офлайн-режим)
        v
[ Сервер: ASP.NET Core + SQLite, self-hosted ]
        |
        v
[ Веб-панель администратора: /admin — таблица сессий, фильтры, CSV-экспорт ]
```

- **`client/`** — трей-приложение на .NET 8 WinForms для Windows. Кнопка
  старт/стоп в трее, иконка-секундомер с бейджем состояния, автозапуск,
  локальная очередь на SQLite (WAL) с автосинком — переживает отсутствие
  интернета и падения процесса. Собирается в один self-contained exe, не
  требует установки .NET на машине сотрудника.
- **`server/`** — ASP.NET Core сервер, принимает события старт/стоп/heartbeat
  от клиентов, пишет их в свою SQLite-базу, отдаёт веб-панель администратора.
  Разворачивается как self-contained exe (Windows/Linux) или в Docker — без
  зависимости от каких-либо облачных сервисов.

## С чего начать

Не хотите ставить .NET SDK и собирать вручную — на странице
[Releases](https://github.com/alximikim/time-tracker-oss/releases) лежат
готовые архивы: клиент под Windows, сервер под Windows и под Linux.
Скачали, распаковали, настроили конфиг (см. ниже) — собирать из исходников
не обязательно.

Есть две роли — обычно нужны обе, но их можно выполнять независимо:

- **Я хочу поднять сервер у себя** (на своём ПК или VPS) →
  [`docs/DEPLOY-SERVER.md`](docs/DEPLOY-SERVER.md)
- **Я хочу собрать и раздать клиента сотрудникам** →
  [`docs/CLIENT-SETUP.md`](docs/CLIENT-SETUP.md)

После разворачивания обеих частей — прогоните
[`docs/VERIFICATION.md`](docs/VERIFICATION.md), чтобы убедиться, что всё
работает целиком: старт/стоп, офлайн-режим, восстановление после сбоя,
реальное выключение компьютера.

## Как это работает

1. Сотрудник включает компьютер — приложение стартует автоматически (трей),
   но **время не считается**, пока не нажата кнопка "Начать работу".
2. Пока сессия активна — тикает локальный таймер, событие уходит в очередь
   и синкается с сервером в фоне.
3. Сессия закрывается вручную кнопкой, либо автоматически при выключении/
   выходе из Windows, либо (после сбоя вроде BSOD/отключения питания) —
   восстанавливается при следующем запуске приложения по последнему
   зафиксированному heartbeat.
4. Если интернета/сервера нет — события копятся локально и досылаются сами,
   как только связь появится. Сотруднику для этого ничего делать не нужно.
5. Администратор смотрит отработанные часы на встроенной веб-панели
   (`/admin` на адресе сервера) — фильтр по сотруднику и датам, итоги,
   экспорт в CSV/Excel.

## Опциональные функции

Включаются администратором один раз в `server-config.json` (см.
[`CLIENT-SETUP.md`](docs/CLIENT-SETUP.md)) — по умолчанию выключены, не
всем компаниям это нужно:

- **Перерыв** (`enableBreakButton`) — кнопка в трее для обеда/отлучек,
  ставит текущую сессию на паузу и возобновляет её позже; итоговые часы за
  день в админ-панели считаются автоматически.
- **Напоминание о длинной сессии** (`longSessionReminderHours`) —
  всплывающее уведомление трея, когда сотрудник отработал за день заданное
  число часов. Только напоминание — ничего не останавливает и не завершает
  за сотрудника.
