# Чеклист проверки (сервер + клиент, локально)

Выполнять по порядку — каждый следующий пункт опирается на то, что
предыдущий прошёл.

## 0. Подготовка

- [ ] `dotnet build` в `server/TimeTracker.Server` и `client/TimeTracker`
      проходит без ошибок.
- [ ] Сервер запущен локально (`dotnet run` в `server/TimeTracker.Server`,
      с `appsettings.Local.json` или переменными окружения — см.
      `DEPLOY-SERVER.md`).
- [ ] В `server-config.json` рядом с клиентским exe стоят `apiUrl`,
      указывающий на локальный сервер (например
      `http://localhost:5140/api/events`), и тот же `sharedToken`, что задан
      на сервере.

## 1. Сервер в изоляции

```powershell
$url = "http://localhost:5140"
Invoke-RestMethod -Uri "$url/api/health"   # {"ok":true,...}
```

- [ ] Health-check возвращает `ok: true`.
- [ ] Тестовый `start` (см. `DEPLOY-SERVER.md`, раздел 4) создаёт сессию.
- [ ] Повторная отправка **того же** `sessionId` с `eventType: start` не
      создаёт вторую строку (идемпотентность) — проверить через `/admin`.

## 2. Первый запуск клиента

- [ ] `dotnet run` из `client/TimeTracker` — форма запрашивает имя, после
      сохранения появляется иконка в трее (серая — «остановлено»).
- [ ] `Get-ItemProperty HKCU:\Software\Microsoft\Windows\CurrentVersion\Run`
      показывает значение `TimeTracker`, указывающее на exe.

## 3. Старт/стоп кнопкой

- [ ] Нажать «Начать работу» → иконка зеленеет, тултип показывает
      «Работаю: 00:00:0X».
- [ ] Тултип считает суммарные часы **за сегодня**, а не только текущий
      отрезок: если сегодня уже была закрытая сессия (например, до
      перерыва или после перезапуска приложения), счётчик сразу стартует
      с накопленного значения, а не с `00:00:00`.
- [ ] Открыть `%AppData%\TimeTracker\data.db` (например, `sqlite3` CLI или
      DB Browser for SQLite) — есть строка `EventType='start'`.
- [ ] В течение ~20-30 секунд строка помечается `Synced=1`, на
      `http://localhost:5140/admin` появляется соответствующая открытая
      сессия (Конец пусто, статус «в процессе»).
- [ ] Нажать «Закончить работу» → иконка сереет, в админ-панели сессия
      закрыта: `Конец` заполнен, `Часов` посчитаны верно, `Статус=кнопка`.

## 4. Офлайн-режим

- [ ] Остановить сервер (Ctrl+C в его консоли) — имитация обрыва связи.
- [ ] Нажать «Начать работу» — событие пишется в `data.db`, но остаётся
      `Synced=0`; в админ-панели (сервер выключен, но представим, что он
      снова поднимется) сессия появится только после синка.
- [ ] Подождать пару минут (проверить, что интервал синка увеличивается —
      бэкофф), снова запустить сервер.
- [ ] Без каких-либо действий пользователя событие само уходит на сервер в
      ближайшем цикле, сессия появляется в `/admin`.

## 5. Восстановление после сбоя

- [ ] Нажать «Начать работу».
- [ ] Жёстко убить процесс: `Stop-Process -Name TimeTracker -Force`
      (эмулирует BSOD/отключение питания — `SessionEnding` не успевает
      сработать).
- [ ] Перезапустить клиента вручную (или перезагрузиться и дать сработать
      автозапуску).
- [ ] Приложение стартует в состоянии «остановлено» (не Running).
- [ ] В `data.db`/в `/admin` сессия закрыта со статусом
      `восстановлено-после-сбоя`, время окончания — примерно момент
      последнего heartbeat (не момент перезапуска).

## 6. Настоящее выключение/выход

- [ ] Нажать «Начать работу».
- [ ] Выполнить обычный «Завершение сеанса» / перезагрузку из меню Пуск.
- [ ] Войти обратно — автозапуск поднимает клиента.
- [ ] Сессия в `/admin` закрыта со статусом `выключение`, время окончания
      примерно соответствует моменту реального выключения.

## 7. Ключевая проверка "opt-in"

- [ ] Просто включить/перезагрузить компьютер, **не нажимая** «Начать
      работу» — ни в `data.db`, ни в `/admin` не появляется ни одной новой
      сессии. Автозапуск сам по себе не создаёт сессию.

## 8. Экспорт CSV

- [ ] Открыть `http://localhost:5140/admin/export.csv` (или кнопку "Экспорт
      CSV" на странице `/admin`) — файл открывается в Excel, кириллица не
      бьётся (файл отдаётся с UTF-8 BOM), дробные часы через точку (`1.25`,
      не `1,25`).
- [ ] Фильтр по сотруднику/датам на `/admin` корректно сужает и таблицу, и
      CSV-экспорт (ссылка экспорта наследует текущие фильтры).

## 9. Перерыв (только если `enableBreakButton: true` в server-config.json)

- [ ] В трее есть пункт "Перерыв", недоступен (disabled) в состоянии
      «остановлено».
- [ ] Нажать «Начать работу», затем «Перерыв» → иконка становится
      оранжевой, тултип «Учёт времени — перерыв», пункт меню меняется на
      «Вернулся с перерыва».
- [ ] В `/admin` за сегодня появилась закрытая сессия со статусом
      `перерыв`.
- [ ] Нажать «Вернулся с перерыва» → создаётся новая сессия (иконка снова
      зелёная), в `/admin` за сегодня — вторая строка на того же сотрудника.
- [ ] Итоговые часы за день в `/admin` — сумма обеих сессий (до и после
      перерыва), без ручного пересчёта.
- [ ] Нажать «Закончить работу» находясь на паузе (не нажимая «Вернулся с
      перерыва») → иконка становится «остановлено», новых строк в `/admin`
      не добавляется (сессия уже была закрыта перерывом).

## 10. Напоминание о длинной сессии (только если задан
    `longSessionReminderHours`)

- [ ] Выставить маленький порог для теста (например, `0.02` — это ~72
      секунды) в `server-config.json`, перезапустить клиента.
- [ ] Нажать «Начать работу», подождать ~2 минуты (heartbeat-таймер тикает
      раз в минуту) — должно появиться всплывающее уведомление трея
      «Вы отработали сегодня уже ... ч».
- [ ] Уведомление не появляется повторно в течение того же дня после того,
      как один раз показалось.
- [ ] Вернуть порог в разумное значение (например, `8`) перед раздачей
      сотрудникам.
